# RFC #0003: Project "Vortex-Transmuter"
**Title:** Deterministic Instruction-Bound Preemption & Unified Transactional Runtime  
**Status:** Proposal / Experimental Development  
**Proposers:** SSL-ACTX (Jameel Tutungan), Iris Seravelle  
**Target Version:** 0.7.0+ (Vortex-X Experimental)

## 1. Abstract
Iris is evolving from a cooperative actor mesh into a **Preemptive Virtual Threading (VT) Fabric**. By moving the scheduling "Heartbeat" from the Rust core directly into the Python bytecode via **In-memory Transmutation**, Iris achieves "Hard" preemption at the instruction level. This RFC introduces the **Vortex-X** extension: a tiered execution model that utilizes transactional I/O buffering and quiescence-gated hot-swapping to solve the "Function Coloring" and "Greedy Actor" problems without sacrificing CPython invariant safety.

---

## 2. The Problem: The "Cooperative Fragility"
Standard Python execution environments suffer from three fatal architectural flaws in high-concurrency systems:
1.  **Reduction Evasion:** Tight loops (e.g., `while True: pass`) bypass Python’s signal checks, allowing a single actor to hijack a CPU core indefinitely.
2.  **The Coloring Tax:** The `async/await` split creates binary code silos, forcing developers to rewrite libraries to support concurrency.
3.  **Side-Effect Leakage:** Speculative or "Ghost" execution often results in "Double-I/O" (e.g., sending two emails) when failing to merge state correctly.

---

## 3. Proposed Solution: The Transmuter Engine

### 3.1. Deterministic Instruction-Bound Preemption (DIBP)
Iris will intercept every `PyCodeObject` during registration. Using a Rust-based bytecode re-assembler, Iris performs **Atomic Injection**:
*   **Injection Points:** Iris identifies all backward-branch targets and function entries.
*   **The Check:** A custom `IRIS_REDUCTION_CHECK` opcode is injected. It performs a single-cycle atomic check against a Rust-owned memory address.
*   **Hard Yield:** If the budget is exhausted, the opcode raises a `VortexSuspend` exception, which is caught by the Rust scheduler to perform a context switch **between bytecode instructions**, ensuring reference-count stability.

### 3.2. Speculative Transactional Parallelism (Ghosting)
To handle grey-failures (slowness) and blocking C-extensions, Iris implements **Transactional Fiber Resurrection**:
*   **Checkpointing:** Iris takes a Copy-on-Write (CoW) snapshot of `f_locals` before high-risk calls.
*   **The Ghost:** If the primary actor stalls, a "Ghost" actor is resurrected from the snapshot to move the Mailbox queue forward.
*   **V-IO (Buffered I/O):** To prevent side-effect duplication, the Ghost actor operates in a **Buffered I/O mode**. All network/system calls are staged in a Rust buffer.
*   **Commit/Abort:** Only the "Winning" actor (the first to reach a safe-point) has its I/O buffer committed to the hardware; the loser's side effects are discarded.

## 4. Technical Architecture: "The Bulletproof Guardrails"

### 4.1. Quiescence-Gated Hot-Swapping (Mid-Await)
To solve the "Mid-Expression Stack Crash" problem, Iris implements **Stack-Depth Validation**:
1.  **Quiescence Point:** The Transmuter identifies instruction offsets where the Evaluation Stack depth is exactly Zero.
2.  **The Swap:** Hot-swapping logic while an actor is `awaiting` is held in a "Staging" state until the actor resumes and hits the next Quiescence Point.
3.  **Holographic Resumption:** The frame is then surgically updated to the new bytecode version, ensuring the "Chef never swaps mid-sandwich."

### 4.2. The "Rescue Pool" (Detached Stalling)
When an actor enters an "Unhinged" stall (e.g., a C-extension that doesn't check for signals):
*   **Isolation:** The Rust core "detaches" the stalled OS thread from the Tokio multi-threaded reactor.
*   **The Rescue:** A new OS thread is spawned to replace it in the Tokio pool, ensuring the system-wide event loop never starves.
*   **Cleanup:** The stalled thread is monitored and eventually reaped or reintegrated once the C-call returns.

## 5. Resilience & Future-Proofing

### 5.1. High-Level Instruction IR
To survive changes in Python 3.13/3.14 (like the Tier 2 JIT or No-GIL), the Transmuter operates on an **Instruction-List IR** rather than raw binary offsets. This abstraction layer ensures that the transmutation logic remains constant even if Python’s binary bytecode format changes.

### 5.2. Vortex Guard (Static Verifier)
Before any bytecode is transmuted, a Rust-based **Vortex Verifier** analyzes the control flow to ensure:
*   Injected opcodes do not disrupt exception-handler stack offsets.
*   `f_stackdepth` invariants are preserved for the specializing interpreter.
*   If a function is too complex for safe transmutation, it falls back to **Cooperative Mode** automatically.

## 6. Absurdity Additions

*   **Genetic Budgeting:** The scheduler dynamically scales an actor's instruction quantum based on its historical yield behavior (punishing "greedy" code by shrinking its CPU time-slice).
*   **Vortex Watchdog:** A `SCHED_FIFO` priority thread in Rust that can trigger `SIGVTALRM` to force-interrupt the VM if bytecode-level preemption is bypassed.
*   **Bytecode-Level Isolation:** Iris rewrites `STORE_GLOBAL` and `STORE_ATTR` opcodes for specific actors to enforce strict memory isolation (enabling "Pure Actor" mode in Python).
